# Anki

Two surfaces:

1. **Anki MCP server** (Anki addon `124672614`, listening on
   `http://127.0.0.1:3141/`) — the authoring channel and the only
   write surface for the agent. Tools include `add_note`/`add_notes`,
   `update_note_fields`, `find_notes`, `notes_info`, `list_decks`,
   `tag_management`, `card_management`, `model_*`, `sync` (triggers
   Anki↔AnkiWeb sync), and review tools.
2. **Cards git repo** — a per-device snapshot of the collection in
   markdown. Generated by `ank_dump.py` (calls MCP), committed by
   `ank_sync.sh`. One file per deck, Q as heading, A as body.
   Read-only from the agent's perspective: never hand-edit deck files.

AnkiWeb handles cross-device card + review state sync separately;
bsrc never touches AnkiWeb credentials.

## When to use
Triggers: "add a card for X", "make Anki cards from this paper",
"extract spaced-rep cards", "snapshot cards to git", "list my Anki
decks", "what tags do I have?", "edit the back of card Y", "find cards
about Z".

## Authoring (write path)

**Before drafting cards, read [`CARD_DESIGN.md`](CARD_DESIGN.md) in
this skill folder.** It owns the atomicity rule, math/figure
conventions, the drafting workflow (batch + review + self-check), tag
conventions, and the knowledge-smith bridge. The rest of this section
is a quick API summary; the design doc is the load-bearing reference.

API summary:

1. Use `list_decks` if unsure which deck to target.
2. Draft 8–20 candidate cards. Present them to the user as a numbered
   list with model, tags, front, and back. **Do not insert yet.**
3. Run the self-check (see CARD_DESIGN §9). Fix or drop failures.
4. After user approves the batch, call `add_notes` (atomic, up to 100)
   with `deckName`, `modelName` (`Basic` or `Cloze`), `fields`, and
   `tags`. For Basic: fields `Front` and `Back`. For Cloze: body in
   `Text` with `{{c1::...}}` syntax, optional explanation in `Back
   Extra`.
5. Suggest `bash ~/.claude/skills/anki/scripts/ank_sync.sh --remote`
   so the new cards land in git.

Bulk-insert without the review step is the most common failure mode in
this skill — don't do it.

## Snapshot (version-control path)
```bash
bash ~/.claude/skills/anki/scripts/ank_sync.sh             # dump + commit
bash ~/.claude/skills/anki/scripts/ank_sync.sh --remote    # dump + commit + push
bash ~/.claude/skills/anki/scripts/ank_sync.sh status      # repo status + MCP probe
```

`ank_dump.py` is invoked by `ank_sync.sh` but can also be run directly:

```bash
uv run ~/.claude/skills/anki/scripts/ank_dump.py
```

It walks all decks via the MCP, writes one .md per deck under the
cards dir, sorted by note id (stable diffs across runs).

## Cards-dir discovery
Scripts find the cards dir by:

1. `$ANKI_CARDS_DIR` env var
2. `~/.config/bsrc/anki.env` (written by `bsrc/anki/install.sh`)
3. Walking cwd upward for the `.anki-cards` marker
4. Hard fail with init hint

There is no default location.

## Health check
```bash
bash ~/.claude/skills/anki/scripts/ank_doctor.sh
```
Reports: Anki app installed, cards dir resolves, marker present,
git origin configured, MCP reachable.

## Don'ts
- Do not hand-edit the deck `.md` files. They are regenerated; edits
  are silently overwritten on the next `ank_sync.sh` run. Author via
  the MCP or via Anki's UI.
- Do not assume Anki is running — `ank_doctor.sh` (or
  `ank_mcp_reachable` in `ank_common.sh`) is the source of truth.
  If the MCP isn't reachable, ask the user to open Anki.
- Do not commit large media binaries you generate — Anki tracks them
  in its collection separately. The deck markdown references them but
  doesn't ship them in this snapshot.
- Do not assume a default cards dir; if discovery fails, tell the user
  to set `ANKI_CARDS_DIR` or re-run `bsrc/anki/install.sh`.

## First-run
After fresh `bsrc/anki/install.sh`:
1. Install Anki MCP addon `124672614` via Anki's Add-ons dialog.
   Restart Anki.
2. Sign in to AnkiWeb in Anki for cross-device sync.
3. Run `bash ~/.claude/skills/anki/scripts/ank_doctor.sh` — should be
   all green.
4. First snapshot: `bash ~/.claude/skills/anki/scripts/ank_sync.sh --remote`.
